# Novel Author

Пишешь ветвящуюся визуальную новеллу под текущую архитектуру проекта. Движок читает
сцены из JSON, поэтому "написать новеллу" = спроектировать граф сцен, накопить
состояние во флагах, развести ветки через условные выборы и довести все пути до
концовок — без багов в связях.

Главный риск не технический, а **потеря контроля над ветвлением**: забытый `next`,
осиротевшая сцена, недостижимая концовка, флаг, который нигде не выставляется. Поэтому
работаешь не "из головы", а от документов-опор, которые создаёшь ПЕРВЫМИ и держишь в
синхроне с JSON до самого конца.

## Точный контракт движка

Прежде чем писать JSON — прочитай `references/engine-contract.md`. Там формат сцены,
семантика `setFlags` (это **присваивание**, не инкремент), что умеет `condition`
(только `&&` и сравнения, нет `||`), как регистрируются сцены и ассеты, и список
ловушек, которые роняют движок. Не пиши сцены по памяти — сверяйся с контрактом, иначе
получишь битый граф.

## Рабочий процесс (от начала до конца, в одной сессии)

Веди TODO-лист по этим фазам. Один `in_progress` за раз. `completed` ставь только после
реальной проверки (сборка/линт/трассировка графа), не по факту намерения.

### Фаза 0 — Бриф

Если пользователь не задал явно, коротко уточни (одним вопросом, с дефолтами):
- **сеттинг и тон** (уютный / драма / детектив / романтика / хоррор…);
- **масштаб** — число сцен, число концовок, глубина ветвления;
- **центральная механика выбора** — какие 1–3 флага копят состояние и что решают
  (симпатия персонажа, найденные улики, храбрость…);
- **персонажи и спрайты** — кто есть, какие выражения нужны.

Не угадывай молча пробелы — подсвети зазор одной строкой и предложи дефолт.

### Фаза 1 — Документы-опоры (создаёшь ДО единого JSON)

Скопируй шаблоны из `assets/` в `docs/novel/` и заполни:

- `docs/novel/BIBLE.md` — библия: логлайн, тон, персонажи, сеттинг, список концовок и
  что каждая означает, "тематический вопрос" истории. Из `assets/BIBLE.template.md`.
- `docs/novel/GRAPH.md` — карта графа: таблица всех сцен (id → куда ведёт), список веток,
  проверка достижимости. Из `assets/GRAPH.template.md`.
- `docs/novel/FLAGS.md` — реестр флагов: имя, тип, где выставляется, где читается,
  диапазон значений, какая концовка каким порогом гейтится. Из `assets/FLAGS.template.md`.

Эти три файла — источник правды. **Любая правка сцен сначала отражается здесь.** Так ты
не теряешься: перед написанием сцены смотришь в GRAPH, кто на неё ссылается и куда она
ведёт; перед условием — смотришь в FLAGS, выставлен ли флаг раньше по пути.

### Фаза 2 — Проектирование графа (на бумаге, до кода)

Стратегия интересного, но управляемого ветвления — в `references/branching-strategy.md`.
Ключевое: используй паттерн **"ветвление и схождение"** (branch-and-bottleneck), а не
чистое дерево. Дерево из N выборов даёт экспоненциальный взрыв сцен и всегда
недоигранные ветки. Вместо этого ветки расходятся, накапливают разницу во флагах и
сходятся в общие узлы — а итоговая концовка выбирается условиями по накопленным флагам.
Разнообразие даёт **состояние (флаги)**, а не количество отдельных сцен.

Зафиксируй в GRAPH.md полный граф ДО написания текста. Проверь на бумаге: каждая сцена
достижима из `start`; у каждой не-концовки есть `next` или `choices`; каждая концовка
достижима хотя бы одним набором выборов; каждый флаг в условии где-то выставляется до
точки чтения.

### Фаза 3 — Написание сцен

Пиши JSON-сцены по контракту. Для каждой сцены:
1. загляни в GRAPH.md (входы/выходы) и FLAGS.md (какие флаги активны на этом пути);
2. напиши сцену: `text` (диалоги/нарратив), нужные `sprites`, `music`/`sfx`, `setFlags`,
   `choices` или `next`, для финалов `isEnding: true`;
3. зарегистрируй сцену в `src/content/scenes/index.ts` (импорт + строка в реестре);
4. новые спрайты/аудио задекларируй в `src/content/manifest.json` (пустой `path` = плейсхолдер по описанию — это нормально, арт можно добавить позже).

Интерактив — не для галочки: условные `choices` (гейт по флагам через `condition`),
`setFlags` для накопления состояния, схождение веток — всё это должно **влиять на
концовку**. Выбор без последствий = мёртвый выбор.

Пиши качественный текст под заданный тон: живые реплики, характер персонажей, крючки
между сценами, чтобы игроку было интересно. См. `references/branching-strategy.md`,
раздел про драматургию выбора.

### Фаза 4 — Верификация (обязательно, до "готово")

Новелла считается доведённой до конца, только когда граф целостен и она собирается:

1. **Трассировка графа** — пройди каждый путь от `start` до концовки, сверяясь с GRAPH.md:
   - каждый `next`/`choice.next` указывает на существующий id в реестре;
   - нет осиротевших сцен (недостижимых из `start`);
   - каждая концовка реально достижима; ни одна не заблокирована навсегда условиями;
   - каждый флаг из `condition` выставляется раньше по пути хотя бы в одной ветке.
2. **Сборка/линт**: `npm run build` и `npm run lint` — должны пройти.
3. Если можешь — прогони игру (`/run` или `npm run dev`) и пройди 1–2 ветки до концовки.

Сообщи итог честно: сколько сцен, сколько концовок, какие ветки прошёл, что проверено.

## Принципы

- **Документы ведут код.** Рассинхрон GRAPH/FLAGS с JSON = ты уже потерялся. Обновляй
  опоры при каждой правке графа.
- **Состояние > количество.** Интерес рождают накопленные флаги и последствия, а не сотни
  линейных сцен.
- **Каждый выбор что-то меняет** — флаг, ветку или тон. Иначе убери выбор.
- **Простота.** Не строй систему флагов сложнее, чем нужно концовкам. YAGNI.
- Всё в одной сессии, без субагентов. Пиши последовательно, держи граф в голове через
  документы-опоры.
